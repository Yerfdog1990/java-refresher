# Log4j 2 Configuration Basics

## 1. Overview

In the previous lesson we used Log4j 2 with its default configuration — ERROR level, minimal format, console output. In this lesson we take full control by writing our own configuration file.

This lesson covers:
- Supported configuration formats and how to choose one
- How Log4j 2 discovers configuration files on the classpath
- The four main elements of a `log4j2.xml` file
- Building and testing a minimal working configuration

---

## 2. Supported Configuration Formats

Log4j 2 supports four configuration formats. All of them expose the same concepts — they differ only in syntax:

| Format | File Name |
|---|---|
| XML | `log4j2.xml` |
| YAML | `log4j2.yaml` / `log4j2.yml` |
| JSON | `log4j2.json` / `log4j2.jsn` |
| Properties | `log4j2.properties` |

Place the file in `src/main/resources` in a standard Maven project — Log4j 2 will find it automatically, no system property required.

**Which format to use?** XML is the most commonly documented and the course default. YAML and JSON are valid alternatives if they better match team preferences or existing project conventions.

---

## 3. Configuration File Discovery

When `LogManager.getLogger()` is first called, Log4j 2 runs a small lookup algorithm to find a configuration file. It scans the classpath in this exact order:

**Test configurations (checked first — preferred during testing):**

```
log4j2-test.properties
log4j2-test.yaml  /  log4j2-test.yml
log4j2-test.json  /  log4j2-test.jsn
log4j2-test.xml
```

**Main configurations (used if no test config is found):**

```
log4j2.properties
log4j2.yaml  /  log4j2.yml
log4j2.json  /  log4j2.jsn
log4j2.xml
```

Log4j 2 uses the **first file it finds** and stops looking. This means placing a `log4j2-test.xml` in `src/test/resources` gives you a verbose test configuration without affecting the production one in `src/main/resources`.

**Override options:** A JVM system property or environment variable can point to an explicit file path. If set, they take precedence over the classpath scan.

**No file found:** Log4j 2 falls back silently to a default configuration — ERROR level, console output — and prints a single status warning.

## 4. Anatomy of `log4j2.xml`

A `log4j2.xml` file has four main parts, all nested inside a single `<Configuration>` root element:

```
<Configuration>
    <Properties>   ← reusable values
    <Appenders>    ← where logs go and how they look
    <Loggers>      ← what gets logged and at what level
        <Root>     ← catch-all baseline
```

We will build each part step by step.

---

### 4.1 `<Configuration>` Root and the `status` Attribute

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Configuration status="WARN">
    <!-- children go here -->
</Configuration>
```

The `status` attribute controls **Log4j 2's own internal logging** during startup — not your application's logging. It is used to diagnose problems in the configuration file itself (typos, invalid references, missing appenders):

| `status` value | What you see |
|---|---|
| `ERROR` | Only critical configuration failures |
| `WARN` *(recommended)* | Important warnings about configuration problems — minimal noise |
| `INFO` | Informational messages about how Log4j 2 loaded the config |
| `DEBUG` / `TRACE` | Detailed step-by-step loading information — use only when debugging config issues |

`status="WARN"` is the best default — it stays quiet during normal operation but surfaces real problems.

---

### 4.2 `<Properties>` — Reusable Values

```xml
<Properties>
    <Property name="logPattern">
        %d{HH:mm:ss} %-5level %logger - %msg%n
    </Property>
</Properties>
```

`<Properties>` defines named values that can be referenced anywhere else in the configuration using `${propertyName}` syntax. This avoids duplicating the same string in multiple places.

#### Pattern placeholder reference

| Placeholder | Meaning | Example output |
|---|---|---|
| `%d{HH:mm:ss}` | Timestamp (hours:minutes:seconds) | `23:58:26` |
| `%-5level` | Log level, left-aligned in a 5-character field | `DEBUG`, `INFO ` |
| `%logger` | Logger name (typically the fully-qualified class name) | `com.baeldung.ljl.JavaLoggingUnitTest` |
| `%msg` | The log message | `Debug message from test` |
| `%n` | Platform-specific newline | (newline) |

#### Environment and system property lookups

Properties can also read values from outside the config file:

```xml
<Properties>
    <Property name="userHome">${env:HOME}</Property>
    <Property name="javaUser">${sys:user.name}</Property>
</Properties>
```

| Lookup prefix | Source |
|---|---|
| `${env:VAR}` | Operating system environment variable |
| `${sys:property}` | JVM system property (`-Dproperty=value`) |

These are useful for building user-specific log file paths or injecting deployment-environment context into log output without hardcoding machine-specific values.

---

### 4.3 `<Appenders>` — Where Logs Go

Appenders define the **destination** for log output. Each appender also contains a layout that defines the **format** of each log line.

```xml
<Appenders>
    <Console name="Console" target="SYSTEM_OUT">
        <PatternLayout pattern="${logPattern}"/>
    </Console>
</Appenders>
```

| Attribute / Element | Value | Meaning |
|---|---|---|
| `name="Console"` | Any unique string | Used to reference this appender from a logger |
| `target="SYSTEM_OUT"` | `SYSTEM_OUT` or `SYSTEM_ERR` | Which output stream to write to |
| `<PatternLayout>` | — | Applies a pattern-based format to each log line |
| `pattern="${logPattern}"` | Property reference | Pulls in the pattern defined in `<Properties>` |

The `name` attribute is how loggers reference appenders — the value itself does not matter as long as it is unique and consistent.

---

### 4.4 `<Loggers>` — What Gets Logged

Loggers decide **which messages are allowed through** based on level, and **where they go** via appender references.

```xml
<Loggers>
    <Root level="DEBUG">
        <AppenderRef ref="Console"/>
    </Root>
</Loggers>
```

| Element / Attribute | Detail |
|---|---|
| `<Root>` | The catch-all logger — applies to every logger in the application unless overridden by a named logger |
| `level="DEBUG"` | The minimum severity level that will be passed to appenders |
| `<AppenderRef ref="Console"/>` | Connects this logger to the appender named `"Console"` |

The separation between `<Loggers>` and `<Appenders>` is intentional — it **decouples** what gets logged from where it goes. You can point the same logger at multiple appenders, or swap appenders without touching the logger configuration.

---

## 5. Complete Minimal `log4j2.xml`

Combining all four sections:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Configuration status="WARN">

    <Properties>
        <Property name="logPattern">
            %d{HH:mm:ss} %-5level %logger - %msg%n
        </Property>
    </Properties>

    <Appenders>
        <Console name="Console" target="SYSTEM_OUT">
            <PatternLayout pattern="${logPattern}"/>
        </Console>
    </Appenders>

    <Loggers>
        <Root level="DEBUG">
            <AppenderRef ref="Console"/>
        </Root>
    </Loggers>

</Configuration>
```

What this configuration does:

- Defines a reusable `logPattern` property
- Creates a `Console` appender that writes to `SYSTEM_OUT` using that pattern
- Sets the root logger to `DEBUG` level, wired to the `Console` appender

## 6. Testing the Configuration

### Test class

```java
class JavaLoggingUnitTest {

    private static final Logger logger = LogManager.getLogger(JavaLoggingUnitTest.class);

    @Test
    void whenLoggingAtDebugAndInfoLevels_thenBothMessagesArePrinted() {
        logger.debug("Debug message from test");
        logger.info("Info message from test");
    }
}
```

### With `<Root level="DEBUG">`

Both messages pass the root logger's level gate and reach the `Console` appender:

```
23:58:26 DEBUG com.baeldung.ljl.JavaLoggingUnitTest - Debug message from test
23:58:26 INFO  com.baeldung.ljl.JavaLoggingUnitTest - Info message from test
```

### With `<Root level="INFO">`

Change only the configuration file — no code change:

```xml
<Root level="INFO">
    <AppenderRef ref="Console"/>
</Root>
```

The `DEBUG` message is now blocked at the root logger's level gate:

```
23:59:46 INFO  com.baeldung.ljl.JavaLoggingUnitTest - Info message from test
```

This demonstrates the core benefit of configuration-driven logging: **changing what appears in the output requires only a config file edit, not a code change**.

---

## 7. How the Four Elements Work Together

```
Java code calls: logger.debug("message")
        ↓
<Root level="DEBUG">         ← is DEBUG ≥ root level? Yes → pass
        ↓
<AppenderRef ref="Console">  ← route to Console appender
        ↓
<PatternLayout>              ← format the message using logPattern
        ↓
target="SYSTEM_OUT"          ← write to console
```

If the root level were `INFO`, the first check would block `DEBUG` and nothing further would happen.

---

## 8. Summary

| Element | Location | Purpose |
|---|---|---|
| `<Configuration status="WARN">` | Root element | Wraps all config; `status` controls Log4j 2's own internal diagnostics |
| `<Properties>` | Inside `<Configuration>` | Defines reusable named values referenced with `${name}` |
| `<Appenders>` | Inside `<Configuration>` | Defines output destinations and their layouts |
| `<Loggers>` / `<Root>` | Inside `<Configuration>` | Sets which levels are allowed and connects loggers to appenders |
| `<PatternLayout pattern="...">` | Inside an appender | Formats each log line using `%` placeholders |
| `<AppenderRef ref="...">` | Inside a logger | Links a logger to a named appender |

**Key takeaways:**
- Place `log4j2.xml` in `src/main/resources` — Log4j 2 finds it automatically, no system property needed.
- Test configs (`log4j2-test.xml` in `src/test/resources`) take priority over main configs, enabling verbose test logging without affecting production.
- Use `<Properties>` to define the log pattern once and reference it with `${logPattern}` to avoid duplication.
- `<Appenders>` and `<Loggers>` are decoupled — loggers reference appenders by name, making it easy to swap destinations without touching logger configuration.
- `status="WARN"` on `<Configuration>` is the recommended setting — quiet in normal operation, visible when configuration has real problems.
- Changing log levels or destinations requires only a config file edit — no code changes needed.
