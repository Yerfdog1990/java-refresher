# Log4j 2 File Appenders

## 1. Overview

Log4j 2 provides three appenders that write log messages to files on the filesystem. They share a common configuration foundation but differ in how they access the file system, which gives each one different performance and concurrency characteristics.

| Appender | File Access Mechanism | Key Characteristic |
|---|---|---|
| `File` | `FileOutputStream` | Standard, reliable, widely compatible |
| `RandomAccessFile` | `RandomAccessFile` | Faster than `File` due to internal buffering |
| `MemoryMappedFile` | `MappedByteBuffer` | Fastest — writes to memory instead of making system calls |

> **Important:** File appenders do not support external tools forcing a log file to be reopened. Tools like `logrotate` must copy the current log file and then truncate it — any log events written during that window will be lost. **If you need automatic log file rotation, use a rolling file appender instead.**

---

## 2. Shared FileManager

When two appenders — even from different logger contexts — use the same `fileName` value, they share a single `FileManager` instance.

```xml
<!-- Both appenders will share one FileManager for "logs/app.log" -->
<File name="AppenderA" fileName="logs/app.log" .../>
<File name="AppenderB" fileName="logs/app.log" .../>
```

**Effect of sharing a FileManager:**
- Guarantees that multiple appenders access the log file **sequentially** — no interleaved or corrupted writes
- Requires most configuration parameters to be identical across the sharing appenders

---

## 3. Common Configuration

All three file appenders share a base set of configuration attributes and nested elements.

### 3.1 Common Attributes

| Attribute | Type | Default | Required | Description |
|---|---|---|---|---|
| `fileName` | Path | — | ✅ Yes | Path to the log file. If the containing folder does not exist, it is created automatically. |
| `name` | String | — | ✅ Yes | Unique name used to reference this appender via `<AppenderRef ref="...">` |
| `bufferSize` | int | `8192` | No | Size in bytes of the internal `ByteBuffer` used before writing to disk |
| `ignoreExceptions` | boolean | `true` | No | If `false`, logging exceptions are forwarded to the caller. If `true` (default), they are silently swallowed. Exceptions are always logged to the Status Logger regardless. |
| `immediateFlush` | boolean | `true` | No | If `true`, the internal buffer is flushed to disk after every log event. Disabling this improves throughput but risks losing recent events on crash. |

### 3.2 Common Nested Elements

| Element | Multiplicity | Description |
|---|---|---|
| `<Filter>` | Zero or one | Filters log events before they are formatted and written — runs at the appender filtering stage |
| `<Layout>` | Zero or one | Defines how log events are formatted before writing. If omitted, a default layout is used. |

## 4. File Appender

The `File` appender uses Java's standard `FileOutputStream` to write to a log file. It is the most straightforward option and the most compatible across operating systems and deployment environments.

### Configuration

```xml
<File name="LogFile"
      fileName="logs/app.log"
      append="true"
      bufferedIo="true"
      immediateFlush="true">
    <PatternLayout pattern="%d{HH:mm:ss} %-5level %logger - %msg%n"/>
</File>
```

### File-Specific Attributes

| Attribute | Type | Default | Description |
|---|---|---|---|
| `append` | boolean | `true` | If `true`, new log entries are added to the end of the existing file. If `false`, the file is overwritten on each application startup. |
| `bufferedIo` | boolean | `true` | If `true`, log events are collected in an internal buffer before being sent to disk. Reduces I/O calls. Works in conjunction with `immediateFlush`. |
| `createOnDemand` | boolean | `false` | If `true`, the file is only created when a log event actually reaches this appender. Useful when a logger may never produce output. |
| `locking` | boolean | `false` | If `true`, Log4j locks the file at each write. Note: most POSIX systems use advisory locking only, not mandatory locking. Enabling this can significantly reduce throughput. |
| `filePermissions` | PosixFilePermissions | `null` | POSIX permission string applied to the created file (e.g. `rw-rw----`). Requires a POSIX-capable filesystem. |
| `fileOwner` | String | `null` | File owner applied to the created file. Requires filesystem support for file owner attributes. |
| `fileGroup` | String | `null` | File group owner applied to the created file. Requires POSIX file attribute view support. |

### `append` vs. `createOnDemand`

| Scenario | Recommended Setting |
|---|---|
| Accumulate logs across application restarts | `append="true"` (default) |
| Start fresh on each restart | `append="false"` |
| Logger may produce no output — avoid creating an empty file | `createOnDemand="true"` |

### `bufferedIo` and `immediateFlush` interaction

| `bufferedIo` | `immediateFlush` | Behavior |
|---|---|---|
| `true` | `true` | Events buffered, then flushed after every event — balanced |
| `true` | `false` | Events buffered, flushed only when buffer is full — highest throughput, risk of data loss on crash |
| `false` | `true` | Every event written directly to disk — safest, lowest throughput |

> **Note:** On most POSIX-compliant systems, `append="true"` guarantees **atomic writes** to the end of the file, even if multiple applications are writing to the same file simultaneously. `RandomAccessFile` and `MemoryMappedFile` do not provide this same guarantee.

## 5. RandomAccessFile Appender

The `RandomAccessFile` appender uses Java's `RandomAccessFile` class instead of `FileOutputStream`. It **always** uses an internal buffer (of size `bufferSize`) regardless of the `bufferedIo` setting, which makes it faster than the standard `File` appender for high-volume logging.

### Configuration

```xml
<RandomAccessFile name="RandomLogFile"
                  fileName="logs/app-random.log"
                  append="true">
    <PatternLayout pattern="%d{HH:mm:ss} %-5level %logger - %msg%n"/>
</RandomAccessFile>
```

### RandomAccessFile-Specific Attributes

| Attribute | Type | Default | Description |
|---|---|---|---|
| `append` | boolean | `true` | If `true`, writing starts at the end of the existing file. |

> **Atomicity note:** Unlike the `File` appender, `RandomAccessFile` does **not** guarantee atomic writes to the end of the file. The log file cannot safely be opened by multiple applications simultaneously.

---

## 6. MemoryMappedFile Appender

The `MemoryMappedFile` appender maps the log file into a `MappedByteBuffer` — a region of memory that the OS keeps synchronized with the underlying file. Writing to this buffer changes program-local memory rather than making system calls to the kernel, which is **orders of magnitude faster** than traditional file I/O.

### Configuration

```xml
<MemoryMappedFile name="MemMapLogFile"
                  fileName="logs/app-memmap.log"
                  append="true"
                  regionLength="33554432">
    <PatternLayout pattern="%d{HH:mm:ss} %-5level %logger - %msg%n"/>
</MemoryMappedFile>
```

### MemoryMappedFile-Specific Attributes

| Attribute | Type | Default | Description |
|---|---|---|---|
| `append` | boolean | `true` | If `true`, writing starts at the end of the existing file. |
| `regionLength` | int | `33554432` (32 MB) | Size in bytes of the memory-mapped buffer. Unlike other file appenders where `bufferSize` is a write buffer, `regionLength` defines the size of the mapped memory region itself. |

> **Atomicity note:** Like `RandomAccessFile`, `MemoryMappedFile` does **not** guarantee atomic writes. The log file cannot safely be opened by multiple applications simultaneously.

> **Trade-off:** The enormous performance gain of memory-mapped I/O comes with a constraint — the `regionLength` memory is reserved upfront. On memory-constrained systems, choose `regionLength` carefully.

---

## 7. Appender Comparison

| Feature | `File` | `RandomAccessFile` | `MemoryMappedFile` |
|---|---|---|---|
| **File access** | `FileOutputStream` | `RandomAccessFile` | `MappedByteBuffer` |
| **Performance** | Standard | Faster | Fastest |
| **Internal buffer** | Optional (`bufferedIo`) | Always (size = `bufferSize`) | Always (size = `regionLength`, default 32 MB) |
| **Atomic multi-app writes** | ✅ Yes (on POSIX) | ✗ No | ✗ No |
| **Multi-app simultaneous access** | ✅ Supported | ✗ Not supported | ✗ Not supported |
| **Best for** | General use, multi-process environments | Single-process, high-throughput | Extreme performance requirements |

---

## 8. When to Use Each Appender

| Scenario | Recommended Appender |
|---|---|
| Standard single-file logging in most applications | `File` |
| High log volume in a single-process application | `RandomAccessFile` |
| Maximum write performance, memory available | `MemoryMappedFile` |
| Log file rotation needed | Use `RollingFile` / `RollingRandomAccessFile` instead |
| External tool (e.g. `logrotate`) manages rotation | Not recommended for any file appender — use rolling variants |

---

## 9. Summary

| Concept | Detail |
|---|---|
| **`File`** | Standard `FileOutputStream`; supports atomic multi-app writes on POSIX; most compatible |
| **`RandomAccessFile`** | Faster; always buffered; single-app access only |
| **`MemoryMappedFile`** | Fastest; writes to memory instead of disk; single-app access only; 32 MB buffer by default |
| **Shared `FileManager`** | Two appenders using the same `fileName` share one manager — sequential access guaranteed |
| **`append`** | `true` = accumulate; `false` = overwrite on startup |
| **`bufferedIo` + `immediateFlush`** | Controls the throughput vs. crash-safety trade-off |
| **`createOnDemand`** | Delays file creation until a log event actually arrives |
| **`locking`** | Advisory file locking on POSIX — reduces performance; use only when necessary |
| **`filePermissions`** | POSIX permission string (e.g. `rw-rw----`); requires POSIX filesystem support |

**Key takeaways:**
- File appenders do not support forced file reopening — external rotation tools like `logrotate` will cause events to be lost during the copy/truncate window. Use rolling file appenders if rotation is needed.
- `File` is the safest choice for compatibility and multi-process environments; `RandomAccessFile` and `MemoryMappedFile` trade those guarantees for higher throughput.
- Shared `FileManager` instances (same `fileName`) guarantee sequential access — useful when multiple appenders legitimately need to write to the same file.
- `immediateFlush="false"` improves performance but risks losing recent log events if the application crashes before the buffer is flushed.
