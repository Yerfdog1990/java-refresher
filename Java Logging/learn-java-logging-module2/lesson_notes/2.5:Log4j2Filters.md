# Log4j 2 Filters

## 1. Overview

Log levels are a blunt instrument — they filter everything below a severity threshold and pass everything above it. Real applications often need more precise control: allow WARN and above but block any message containing a credit card number; throttle a noisy logger that fires hundreds of times per second; silently drop any message containing the word "password".

**Filters** are Log4j 2's answer to these requirements. They evaluate log events against custom criteria and make independent accept/deny decisions, going far beyond what level-based filtering alone can do.

This lesson covers:
- How filters work and where they can be placed
- `ThresholdFilter` — level-based filtering with fine-grained control
- `BurstFilter` — rate limiting to prevent log flooding
- `RegexFilter` — content-based filtering using regular expressions
- Combining multiple filters into a chain

---

## 2. How Filters Work

### Filter Decisions

Every filter returns one of three decisions for each log event:

| Decision | Meaning |
|---|---|
| `ACCEPT` | Process the log event immediately — skip all remaining filters |
| `DENY` | Discard the log event immediately — skip all remaining filters |
| `NEUTRAL` | Pass the event to the next filter in the chain for a decision |

If all filters in a chain return `NEUTRAL`, Log4j 2 falls back to the normal logger/appender configuration (log level, additivity rules, etc.) to decide whether to log the event.

### Where Filters Can Be Placed

| Scope | Location in Config | What It Controls |
|---|---|---|
| **Global** | Top-level `<Filters>` | All log events before they reach any logger |
| **Logger** | Inside `<Logger>` | Which events a specific logger passes to its appenders |
| **Appender** | Inside `<Appender>` | Output of an individual appender |
| **Appender Reference** | Inside `<AppenderRef>` | Fine-tuned control when one logger writes to multiple appenders |

This lesson uses **Appender Filters** — they provide the most granular, per-destination control and are the most common placement in practice.

## 3. ThresholdFilter

`ThresholdFilter` filters log events based on a configured severity level. Unlike setting a level directly on a logger or appender, `ThresholdFilter` gives independent `onMatch` and `onMismatch` decisions and can be combined with other filters in a chain.

### Configuration

```xml
<Appenders>
    <Console name="ThresholdFilterConsole" target="SYSTEM_OUT">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} %-5p %c{1.} - %m%n"/>
        <ThresholdFilter level="ERROR" onMatch="ACCEPT" onMismatch="DENY"/>
    </Console>
</Appenders>
<Loggers>
    <Logger name="com.baeldung.ljl.filters.threshold" level="DEBUG" additivity="false">
        <AppenderRef ref="ThresholdFilterConsole"/>
    </Logger>
</Loggers>
```

| Attribute | Value | Meaning |
|---|---|---|
| `level` | `ERROR` | The severity threshold to compare against |
| `onMatch` | `ACCEPT` | If the event level is ERROR or above — accept immediately |
| `onMismatch` | `DENY` | If the event level is below ERROR — discard immediately |

Note that the **logger** is configured at `DEBUG` level. Without the filter, DEBUG, INFO, WARN, ERROR, and FATAL would all reach the appender. The `ThresholdFilter` intercepts at the appender level and enforces its own stricter rule.

### Test

```java
@Test
void whenUsingLoggerWithThresholdFilter_thenLogsFilteredBasedOnConfiguredLevel() {
    Logger logger = LogManager.getLogger("com.baeldung.ljl.filters.threshold");

    logger.debug("Sample DEBUG log");
    logger.info("Sample INFO log");
    logger.warn("Sample WARN log");
    logger.error("Sample ERROR log");
    logger.fatal("Sample FATAL log");
}
```

### Output

```
2025-09-20 18:30:02 ERROR c.b.l.f.threshold - Sample ERROR log
2025-09-20 18:30:02 FATAL c.b.l.f.threshold - Sample FATAL log
```

DEBUG, INFO, and WARN are blocked by the filter even though the logger itself allows them. This is the key separation of concerns — the logger decides what enters the pipeline, the filter decides what the appender actually writes.

### ThresholdFilter vs. Setting Level on AppenderRef

| | `level` on `<AppenderRef>` | `ThresholdFilter` |
|---|---|---|
| Blocks below threshold | Yes | Yes |
| Independent `onMatch` / `onMismatch` control | No | Yes |
| Can be chained with other filters | No | Yes |
| Suitable for complex multi-filter policies | No | Yes |

`ThresholdFilter` is the better choice whenever you need to compose it with other filters or apply different `onMatch`/`onMismatch` decisions in a chain.

## 4. BurstFilter

Some events occur in rapid bursts — a retry loop firing hundreds of times, an exception being swallowed in a tight loop. Without rate limiting, these events flood the logs with redundant, nearly identical entries. `BurstFilter` solves this using a **sliding-window algorithm**.

### How BurstFilter Works

`BurstFilter` maintains a sliding time window defined by two parameters:

| Parameter | Default | Description |
|---|---|---|
| `rate` | `10` | Expected average number of messages per second |
| `maxBurst` | `10 × rate` | Maximum number of messages allowed within the sliding window |

The **window size** in seconds = `maxBurst / rate`.

**Example:** `rate=5`, `maxBurst=10`:
- Window size = `10 / 5` = **2 seconds**
- Up to 10 events are allowed within any rolling 2-second window
- When the 11th event arrives and 10 events already exist in the last 2 seconds → **DENY**
- As time advances and older events fall outside the 2-second window, capacity gradually returns

### Configuration

```xml
<Appenders>
    <Console name="BurstFilterConsole" target="SYSTEM_OUT">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} %-5p %c{1.} - %m%n"/>
        <BurstFilter level="INFO" rate="5" maxBurst="10"/>
    </Console>
</Appenders>
<Loggers>
    <Logger name="com.baeldung.ljl.filters.burst" level="DEBUG" additivity="false">
        <AppenderRef ref="BurstFilterConsole"/>
    </Logger>
</Loggers>
```

| Attribute | Value | Meaning |
|---|---|---|
| `level` | `INFO` | Only applies rate limiting to events at INFO level and above |
| `rate` | `5` | Sustained rate of 5 messages per second |
| `maxBurst` | `10` | Up to 10 messages allowed in a single burst before throttling |

### Test

```java
@Test
void whenUsingLoggerWithBurstFilter_thenLogsRateLimitedAfterBurst() throws InterruptedException {
    Logger logger = LogManager.getLogger("com.baeldung.ljl.filters.burst");

    for (int i = 1; i <= 100; i++) {
        logger.info("Burst message #" + i);
    }

    System.out.println("\n[Waiting for 2 seconds]\n");
    Thread.sleep(2000);

    for (int i = 1; i <= 100; i++) {
        logger.info("Post waiting burst message #" + i);
    }
}
```

### Output

```
2025-09-20 18:30:33 INFO  c.b.l.f.burst - Burst message #1
2025-09-20 18:30:33 INFO  c.b.l.f.burst - Burst message #2
...
2025-09-20 18:30:33 INFO  c.b.l.f.burst - Burst message #10

[Waiting for 2 seconds]

2025-09-20 18:30:35 INFO  c.b.l.f.burst - Post waiting burst message #1
2025-09-20 18:30:35 INFO  c.b.l.f.burst - Post waiting burst message #2
...
2025-09-20 18:30:35 INFO  c.b.l.f.burst - Post waiting burst message #10
```

100 messages were sent in rapid succession. Only the first 10 appear — messages 11 through 100 hit the burst limit and were denied. After the 2-second wait, the window resets and the next burst of 100 again yields 10 messages.

## 5. RegexFilter

`RegexFilter` matches the content of the log message against a regular expression. This enables **content-based filtering** — independent of level or frequency. A common use case is blocking messages that contain sensitive data such as passwords, tokens, or credit card numbers before they reach a log destination.

### Configuration

```xml
<Appenders>
    <Console name="RegexFilterConsole" target="SYSTEM_OUT">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} %-5p %c{1.} - %m%n"/>
        <RegexFilter regex=".*(?i)password.*" onMatch="DENY" onMismatch="ACCEPT"/>
    </Console>
</Appenders>
<Loggers>
    <Logger name="com.baeldung.ljl.filters.regex" level="DEBUG" additivity="false">
        <AppenderRef ref="RegexFilterConsole"/>
    </Logger>
</Loggers>
```

| Attribute | Value | Meaning |
|---|---|---|
| `regex` | `.*(?i)password.*` | Matches any message containing "password" (case-insensitive via `(?i)`) |
| `onMatch` | `DENY` | If the pattern matches — discard the event |
| `onMismatch` | `ACCEPT` | If the pattern does not match — allow the event through |

### Test

```java
@Test
void whenUsingLoggerWithRegexFilter_thenLogsFilteredBasedOnPattern() {
    Logger logger = LogManager.getLogger("com.baeldung.ljl.filters.regex");

    logger.warn("Invalid login attempt - incorrect password entered");
    logger.info("Initiating password reset workflow");
    logger.error("New password cannot be same as previous");
}
```

### Output

```
(no output)
```

All three messages contain the word "password" — the `RegexFilter` blocks all of them regardless of severity level. The console receives nothing.

## 6. Combining Multiple Filters

When filtering requirements involve multiple criteria, Log4j 2 allows multiple filters to be chained inside a `<Filters>` wrapper. They are evaluated **in the order they are defined**. The first filter to return `ACCEPT` or `DENY` ends the chain immediately — only `NEUTRAL` passes control to the next filter.

### Scenario: Block Low-Severity Events and Redact Credit Card Numbers

```xml
<Appenders>
    <Console name="CombinedFiltersConsole" target="SYSTEM_OUT">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} %-5p %c{1.} - %m%n"/>
        <Filters>
            <ThresholdFilter level="WARN"
                             onMatch="NEUTRAL"
                             onMismatch="DENY"/>
            <RegexFilter regex=".*\b4[0-9]{12}(?:[0-9]{3})?\b.*"
                         onMatch="DENY"
                         onMismatch="NEUTRAL"/>
            <RegexFilter regex=".*\b5[1-5][0-9]{14}\b.*"
                         onMatch="DENY"
                         onMismatch="ACCEPT"/>
        </Filters>
    </Console>
</Appenders>
<Loggers>
    <Logger name="com.baeldung.ljl.filters.combined" level="DEBUG" additivity="false">
        <AppenderRef ref="CombinedFiltersConsole"/>
    </Logger>
</Loggers>
```

### Filter Chain Walkthrough

| # | Filter | `onMatch` | `onMismatch` | Rule |
|---|---|---|---|---|
| 1 | `ThresholdFilter level="WARN"` | `NEUTRAL` | `DENY` | Block anything below WARN; pass WARN+ to next filter |
| 2 | `RegexFilter` (Visa pattern) | `DENY` | `NEUTRAL` | Block Visa card numbers; pass others to next filter |
| 3 | `RegexFilter` (Mastercard pattern) | `DENY` | `ACCEPT` | Block Mastercard numbers; accept everything else |

The regex patterns used:

| Pattern | Matches |
|---|---|
| `\b4[0-9]{12}(?:[0-9]{3})?\b` | Visa card numbers (13 or 16 digits starting with 4) |
| `\b5[1-5][0-9]{14}\b` | Mastercard numbers (16 digits starting with 51–55) |

### Decision Flow

```
Event arrives at filter chain
         ↓
[1] ThresholdFilter: level < WARN?
         → DENY (dropped)    ← DEBUG, INFO blocked here
         → NEUTRAL           ← WARN, ERROR, FATAL continue
         ↓
[2] RegexFilter (Visa): message matches Visa pattern?
         → DENY (dropped)    ← Visa card numbers blocked here
         → NEUTRAL           ← others continue
         ↓
[3] RegexFilter (Mastercard): message matches Mastercard pattern?
         → DENY (dropped)    ← Mastercard numbers blocked here
         → ACCEPT            ← safe messages reach the appender
```

### Test

```java
@Test
void whenUsingLoggerWithCombinedFilters_thenLogsFilteredThroughAllFilters() {
    Logger logger = LogManager.getLogger("com.baeldung.ljl.filters.combined");

    logger.warn("Suspicious payment detected with card: 4532015112830366");  // Visa
    logger.error("Failed transaction for card: 5425233430109903");           // Mastercard
    logger.warn("Transaction retry limit exceeded");                          // Safe
}
```

### Output

```
2025-09-20 18:32:38 WARN  c.b.l.f.combined - Transaction retry limit exceeded
```

| Message | ThresholdFilter | Visa RegexFilter | Mastercard RegexFilter | Result |
|---|---|---|---|---|
| `warn("...card: 4532015112830366")` | NEUTRAL (WARN ≥ WARN) | DENY (Visa match) | — | **Dropped** |
| `error("...card: 5425233430109903")` | NEUTRAL (ERROR ≥ WARN) | NEUTRAL (no Visa) | DENY (Mastercard match) | **Dropped** |
| `warn("Transaction retry...")` | NEUTRAL (WARN ≥ WARN) | NEUTRAL (no Visa) | ACCEPT (no Mastercard) | **Logged** |

Only the safe message makes it through. The credit card numbers are redacted at the logging layer before they could ever reach any storage destination.

## 7. Complete `log4j2.xml`

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Configuration status="WARN">

    <Appenders>

        <!-- ThresholdFilter: only ERROR and FATAL -->
        <Console name="ThresholdFilterConsole" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} %-5p %c{1.} - %m%n"/>
            <ThresholdFilter level="ERROR" onMatch="ACCEPT" onMismatch="DENY"/>
        </Console>

        <!-- BurstFilter: max 10 messages per 2-second window -->
        <Console name="BurstFilterConsole" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} %-5p %c{1.} - %m%n"/>
            <BurstFilter level="INFO" rate="5" maxBurst="10"/>
        </Console>

        <!-- RegexFilter: block messages containing "password" -->
        <Console name="RegexFilterConsole" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} %-5p %c{1.} - %m%n"/>
            <RegexFilter regex=".*(?i)password.*" onMatch="DENY" onMismatch="ACCEPT"/>
        </Console>

        <!-- Combined filters: WARN+, no credit card numbers -->
        <Console name="CombinedFiltersConsole" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} %-5p %c{1.} - %m%n"/>
            <Filters>
                <ThresholdFilter level="WARN" onMatch="NEUTRAL" onMismatch="DENY"/>
                <RegexFilter regex=".*\b4[0-9]{12}(?:[0-9]{3})?\b.*" onMatch="DENY" onMismatch="NEUTRAL"/>
                <RegexFilter regex=".*\b5[1-5][0-9]{14}\b.*" onMatch="DENY" onMismatch="ACCEPT"/>
            </Filters>
        </Console>

    </Appenders>

    <Loggers>
        <Root level="DEBUG">
            <AppenderRef ref="ThresholdFilterConsole"/>
        </Root>
        <Logger name="com.baeldung.ljl.filters.threshold" level="DEBUG" additivity="false">
            <AppenderRef ref="ThresholdFilterConsole"/>
        </Logger>
        <Logger name="com.baeldung.ljl.filters.burst" level="DEBUG" additivity="false">
            <AppenderRef ref="BurstFilterConsole"/>
        </Logger>
        <Logger name="com.baeldung.ljl.filters.regex" level="DEBUG" additivity="false">
            <AppenderRef ref="RegexFilterConsole"/>
        </Logger>
        <Logger name="com.baeldung.ljl.filters.combined" level="DEBUG" additivity="false">
            <AppenderRef ref="CombinedFiltersConsole"/>
        </Logger>
    </Loggers>

</Configuration>
```

## 8. Summary

| Filter | Filters On | Common Use Case |
|---|---|---|
| `ThresholdFilter` | Severity level | Allow only ERROR+ to a specific appender while the logger remains at DEBUG |
| `BurstFilter` | Message rate | Prevent log flooding from retry loops or high-frequency events |
| `RegexFilter` | Message content | Block sensitive data (passwords, tokens, card numbers) from log destinations |

| Concept | Detail |
|---|---|
| **Filter decisions** | `ACCEPT` (stop chain, allow), `DENY` (stop chain, discard), `NEUTRAL` (pass to next filter) |
| **Filter placement** | Global, Logger, Appender, or AppenderRef — most granular at Appender level |
| **`<Filters>` wrapper** | Required when combining multiple filters on one appender; evaluated in declaration order |
| **First terminal decision wins** | `ACCEPT` or `DENY` stops the chain; only `NEUTRAL` continues |
| **`onMatch` / `onMismatch`** | Independent control over the matched and unmatched cases |

**Key takeaways:**
- Filters operate independently of log levels — a logger at DEBUG can have an appender that only receives ERROR events via `ThresholdFilter`.
- `ThresholdFilter` is more powerful than setting `level` on `<AppenderRef>` because it supports `onMatch`/`onMismatch` and can be chained.
- `BurstFilter` is essential for preventing log flooding from high-frequency events — configure `rate` and `maxBurst` based on your expected sustained vs. peak log volume.
- `RegexFilter` with `onMatch="DENY"` is a simple but effective last line of defense for keeping sensitive data out of log files.
- When chaining filters, use `NEUTRAL` on `onMatch` or `onMismatch` to pass control to the next filter — a terminal `ACCEPT` or `DENY` always ends the chain.
