# Log4j 2 Appenders and Layouts

## 1. Overview

In the previous lesson we built a minimal `log4j2.xml` with a single Console appender. This lesson goes deeper — configuring multiple appenders, controlling output format with layouts, and injecting contextual data into log messages using MDC.

This lesson covers:
- What appenders and layouts are and how they relate
- Console, File, and RollingFile appenders
- PatternLayout and JSONTemplateLayout
- Mapped Diagnostic Context (MDC) for per-request contextual logging

---

## 2. Appenders and Layouts

### Appenders

An **appender** is a component that sends log messages to a specific destination — console, file, database, remote network service, message queue, and so on.

Key behaviors:
- A **single logger** can write to **multiple appenders** simultaneously
- **Multiple loggers** can share the **same appender**
- Loggers **inherit appenders** from their parents in the logger hierarchy — an appender attached to the root logger is automatically available to all loggers in the application

### Layouts

A **layout** (called a *formatter* in some other frameworks) controls how a log event is formatted before it is written. Each appender has exactly one layout associated with it.

```
Logger → decides what gets logged and at what level
   ↓
Appender → decides where it goes
   ↓
Layout → decides how it looks
```

This separation means you can change the format without changing the destination, and change the destination without changing the format.

## 3. Appenders

### 3.1 Console Appender

The `Console` appender writes log messages directly to the JVM's standard I/O streams — either `System.out` or `System.err`.

```xml
<Configuration status="WARN">
    <Appenders>
        <Console name="Console" target="SYSTEM_OUT">
        </Console>
    </Appenders>
    <Loggers>
        <Root level="INFO">
            <AppenderRef ref="Console"/>
        </Root>
    </Loggers>
</Configuration>
```

| Attribute | Value | Meaning |
|---|---|---|
| `name` | `"Console"` | Unique identifier — used in `<AppenderRef ref="...">` |
| `target` | `SYSTEM_OUT` | Writes to `System.out`; use `SYSTEM_ERR` for `System.err` |

**Test:**

```java
class JavaLoggingUnitTest {
    private static final Logger logger = LogManager.getLogger(JavaLoggingUnitTest.class);

    @Test
    void whenMessagesLoggedUsingRootLogger_thenMessagesPrintedToConsole() {
        logAtAllLevels(logger);
    }

    private void logAtAllLevels(Logger logger) {
        logger.trace("Sample TRACE log");
        logger.debug("Sample DEBUG log");
        logger.info("Sample INFO log");
        logger.warn("Sample WARN log");
        logger.error("Sample ERROR log");
        logger.fatal("Sample FATAL log");
    }
}
```

**Output:**

```
Sample INFO log
Sample WARN log
Sample ERROR log
Sample FATAL log
```

Only INFO and above appear — the root logger's `level="INFO"` blocks TRACE and DEBUG. The output is plain text with no metadata because no layout has been configured yet. We will fix this in Section 4.

> The `logAtAllLevels()` helper method is extracted for reuse across subsequent tests in this lesson.

### 3.2 File Appender

The `File` appender writes log messages to a file on the filesystem — useful for persisting logs for later analysis and auditing.

```xml
<Appenders>
    <!-- existing Console appender -->
    <File name="LogFile" fileName="logs/app.log" append="true">
    </File>
</Appenders>
<Loggers>
    <!-- existing Root logger -->
    <Logger name="LoggerWithFileAppender" level="INFO" additivity="false">
        <AppenderRef ref="LogFile"/>
    </Logger>
</Loggers>
```

| Attribute / Element | Value | Meaning |
|---|---|---|
| `fileName` | `"logs/app.log"` | Path to the log file; the `logs/` directory is created automatically |
| `append="true"` | `true` | New log entries are added to the end of the file; `false` would overwrite the file on every startup |
| `name="LoggerWithFileAppender"` | Named logger | Obtained via `LogManager.getLogger("LoggerWithFileAppender")` |
| `additivity="false"` | Disabled | Prevents log events from bubbling up to parent loggers (see below) |

#### Understanding `additivity`

By default, log events propagate **up** the logger hierarchy to parent loggers and ultimately to the root logger. This means without `additivity="false"`, a message logged via `LoggerWithFileAppender` would appear in **both** the file (via `LogFile`) and the console (via the root logger's `Console` appender).

```
additivity="true" (default):
  LoggerWithFileAppender → LogFile appender
                         ↑ also bubbles up to root
                         → Console appender too

additivity="false":
  LoggerWithFileAppender → LogFile appender only
```

Setting `additivity="false"` isolates the logger to its own appenders only.

**Test:**

```java
@Test
void whenMessagesLoggedUsingLoggerWithFileAppender_thenMessagesWrittenToFile() {
    Logger loggerWithFileAppender = LogManager.getLogger("LoggerWithFileAppender");
    logAtAllLevels(loggerWithFileAppender);
}
```

After running, a `logs/app.log` file appears in the project root containing the same messages that would otherwise have gone to the console.

### 3.3 RollingFile Appender

A plain `File` appender writes everything to a single file forever. For long-running applications that generate high log volumes, this file grows unmanageably large. The `RollingFile` appender solves this by automatically archiving the current log file and starting a new one based on configurable criteria.

```xml
<Appenders>
    <!-- existing appenders -->
    <RollingFile name="RollingFile"
                 fileName="rolling-logs/app.log"
                 filePattern="rolling-logs/app-%i.log">
        <Policies>
            <SizeBasedTriggeringPolicy size="10KB"/>
        </Policies>
    </RollingFile>
</Appenders>
<Loggers>
    <!-- existing loggers -->
    <Logger name="LoggerWithRollingFileAppender" level="INFO" additivity="false">
        <AppenderRef ref="RollingFile"/>
    </Logger>
</Loggers>
```

| Attribute / Element | Value | Meaning |
|---|---|---|
| `fileName` | `"rolling-logs/app.log"` | The active log file currently being written to |
| `filePattern` | `"rolling-logs/app-%i.log"` | Naming pattern for archived files; `%i` is an auto-incrementing counter |
| `<SizeBasedTriggeringPolicy size="10KB">` | 10KB | Roll the file when it reaches 10KB |

#### filePattern placeholders

| Placeholder | Meaning | Example |
|---|---|---|
| `%i` | Auto-incrementing counter | `app-1.log`, `app-2.log`, `app-3.log` |
| `%d{yyyy-MM-dd}` | Current date | `app-2025-09-24.log` |

> Log4j 2 provides additional triggering policies beyond size — including time-based (`TimeBasedTriggeringPolicy`) and composite (`CronTriggeringPolicy`). See the official documentation for the full list.

**Test:**

```java
@Test
void whenMessagesLoggedUsingLoggerWithRollingFileAppender_thenMessagesWrittenToFiles() {
    Logger loggerWithRollingFileAppender = LogManager.getLogger("LoggerWithRollingFileAppender");
    for (int i = 0; i < 1000; i++) {
        logAtAllLevels(loggerWithRollingFileAppender);
    }
}
```

After running, the `rolling-logs/` directory contains multiple files following the configured naming pattern — `app.log` (the active file) and `app-1.log`, `app-2.log`, etc. (archived files).

---

### Appender Comparison

| Appender | Destination | Best For |
|---|---|---|
| `Console` | `System.out` / `System.err` | Development and debugging |
| `File` | Single file | Simple persistence, auditing |
| `RollingFile` | Multiple files (auto-rotated) | Production — prevents unbounded file growth |

## 4. Layouts

### 4.1 PatternLayout

`PatternLayout` formats each log line using a pattern string containing conversion characters that Log4j 2 replaces with actual values at runtime.

```xml
<Console name="Console" target="SYSTEM_OUT">
    <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%t] %-5p %c - %m%n"/>
</Console>
```

#### Conversion character reference

| Character | Meaning | Example output |
|---|---|---|
| `%d{yyyy-MM-dd HH:mm:ss}` | Date and time | `2025-09-24 21:43:36` |
| `%t` | Thread name | `main` |
| `%-5p` | Log level, left-aligned in a 5-character field | `INFO `, `WARN `, `ERROR` |
| `%c` | Logger name (fully-qualified class name) | `com.baeldung.ljl.JavaLoggingUnitTest` |
| `%m` | Log message | `Sample INFO log` |
| `%n` | Platform-specific newline | (newline) |

> `%-5p` uses `-` for left-alignment and `5` as the minimum field width. This keeps log levels visually aligned across lines of different level name lengths.

**Output with PatternLayout:**

```
2025-09-24 21:43:36 [main] INFO  com.baeldung.ljl.JavaLoggingUnitTest - Sample INFO log
2025-09-24 21:43:36 [main] WARN  com.baeldung.ljl.JavaLoggingUnitTest - Sample WARN log
2025-09-24 21:43:36 [main] ERROR com.baeldung.ljl.JavaLoggingUnitTest - Sample ERROR log
2025-09-24 21:43:36 [main] FATAL com.baeldung.ljl.JavaLoggingUnitTest - Sample FATAL log
```

Compare this to the plain output from Section 3.1 — adding a `PatternLayout` transforms unreadable bare messages into fully contextualized, aligned log lines without changing any Java code.

### 4.2 JSONTemplateLayout

While `PatternLayout` produces human-readable text, modern log aggregation and analysis platforms (ELK Stack, Splunk, Datadog, etc.) prefer structured data. `JSONTemplateLayout` formats each log event as a self-contained JSON object.

`JSONTemplateLayout` is not part of the core `log4j-core` library — it requires an additional dependency:

```xml
<dependency>
    <groupId>org.apache.logging.log4j</groupId>
    <artifactId>log4j-layout-template-json</artifactId>
    <version>${log4j.version}</version>
</dependency>
```

**Configuration:**

```xml
<Appenders>
    <!-- existing appenders -->
    <File name="JsonLogFile" fileName="json-logs/app.json" append="true">
        <JSONTemplateLayout/>
    </File>
</Appenders>
<Loggers>
    <!-- existing loggers -->
    <Logger name="LoggerWithJsonFileAppender" level="INFO" additivity="false">
        <AppenderRef ref="JsonLogFile"/>
    </Logger>
</Loggers>
```

**Test:**

```java
@Test
void whenMessagesLoggedUsingLoggerWithJsonFileAppender_thenMessagesWrittenToJsonFile() {
    Logger loggerWithJsonFileAppender = LogManager.getLogger("LoggerWithJsonFileAppender");
    logAtAllLevels(loggerWithJsonFileAppender);
}
```

**Output in `json-logs/app.json`:**

```json
{"@timestamp":"2025-09-24T16:22:24.390Z","ecs.version":"1.2.0","log.level":"INFO","message":"Sample INFO log","process.thread.name":"main","log.logger":"LoggerWithJsonFileAppender"}
{"@timestamp":"2025-09-24T16:22:24.391Z","ecs.version":"1.2.0","log.level":"WARN","message":"Sample WARN log","process.thread.name":"main","log.logger":"LoggerWithJsonFileAppender"}
{"@timestamp":"2025-09-24T16:22:24.392Z","ecs.version":"1.2.0","log.level":"ERROR","message":"Sample ERROR log","process.thread.name":"main","log.logger":"LoggerWithJsonFileAppender"}
{"@timestamp":"2025-09-24T16:22:24.392Z","ecs.version":"1.2.0","log.level":"FATAL","message":"Sample FATAL log","process.thread.name":"main","log.logger":"LoggerWithJsonFileAppender"}
```

Each line is a complete, independently parseable JSON object. The layout automatically includes ECS (Elastic Common Schema) fields — `@timestamp`, `ecs.version`, `log.level`, `message`, `process.thread.name`, `log.logger` — which log aggregation systems can directly ingest without additional parsing configuration.

#### Layout Comparison

| Layout | Format | Best For |
|---|---|---|
| `PatternLayout` | Plain text | Human reading; console output; simple log files |
| `JSONTemplateLayout` | Structured JSON | Log aggregation tools (ELK, Splunk, Datadog); machine parsing |

## 5. Mapped Diagnostic Context (MDC)

### What Is MDC?

In a multi-user web application, many requests are processed concurrently by different threads. When an error occurs, it can be very difficult to correlate all the log entries belonging to a single request — they are interleaved with log entries from other requests.

**MDC (Mapped Diagnostic Context)** solves this by providing a **thread-local map** where you can store key-value pairs. Log4j 2 automatically includes these values in every log message produced by that thread — without you having to pass them to every log call.

```
ThreadContext.put("userId", "12345")
     ↓
Every subsequent log call on this thread automatically includes userId=12345
```

MDC values are stored **per thread** — they do not bleed into other threads. In asynchronous or multi-threaded applications, they may need to be propagated explicitly.

### Configuration

Add MDC placeholders to the `PatternLayout` using `%X{key}` or its alias `%mdc{key}`:

```xml
<Console name="ConsoleWithMDC" target="SYSTEM_OUT">
    <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%t] %-5p %c [%X{userName} - %mdc{userId}] - %m%n"/>
</Console>

<Logger name="LoggerWithMDC" level="INFO" additivity="false">
    <AppenderRef ref="ConsoleWithMDC"/>
</Logger>
```

| Placeholder | Alias | Meaning |
|---|---|---|
| `%X{key}` | `%mdc{key}` | Retrieves the value for `key` from the MDC; empty if not set |

### Populating the MDC

```java
@Test
void whenMessagesLoggedUsingLoggerWithMDC_thenMessagesContainContextData() {
    Logger loggerWithMDC = LogManager.getLogger("LoggerWithMDC");

    loggerWithMDC.info("Initiating withdrawal request");  // MDC is empty

    ThreadContext.put("userName", "John Doe");
    ThreadContext.put("userId", UUID.randomUUID().toString());

    loggerWithMDC.info("Initiating withdrawal request");  // MDC is populated
}
```

**Output:**

```
2025-09-24 21:57:28 [main] INFO  LoggerWithMDC [ - ] - Initiating withdrawal request
2025-09-24 21:57:28 [main] INFO  LoggerWithMDC [John Doe - 5d9fd667-ad31-46e5-80b3-b9144691d9db] - Initiating withdrawal request
```

| Line | MDC state | Output for `[%X{userName} - %mdc{userId}]` |
|---|---|---|
| First log call | Empty | `[ - ]` — placeholders render as blank |
| After `ThreadContext.put()` calls | Populated | `[John Doe - 5d9fd667-...]` — values injected automatically |

The same `info()` call produces different output depending on what is in the MDC at that moment — no change to the log call itself is needed.

### Lookups in Configuration

Log4j 2 also supports referencing external values directly in the configuration file via a **Lookup** mechanism:

| Lookup | Syntax | Example |
|---|---|---|
| System property | `${sys:property}` | `${sys:user.home}`, `${sys:java.version}` |
| Environment variable | `${env:VAR}` | `${env:PATH}`, `${env:HOME}` |
| Log4j 2 internals | `${log4j:key}` | `${log4j:configLocation}` |

These are useful for building dynamic file paths, avoiding hardcoded machine-specific values, and adding deployment context to log output.

## 6. Complete `log4j2.xml`

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Configuration status="WARN">

    <Appenders>

        <!-- Console: human-readable text for development -->
        <Console name="Console" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%t] %-5p %c - %m%n"/>
        </Console>

        <!-- Console with MDC: includes per-request context data -->
        <Console name="ConsoleWithMDC" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%t] %-5p %c [%X{userName} - %mdc{userId}] - %m%n"/>
        </Console>

        <!-- File: simple persistent log -->
        <File name="LogFile" fileName="logs/app.log" append="true">
        </File>

        <!-- RollingFile: auto-rotated persistent log -->
        <RollingFile name="RollingFile"
                     fileName="rolling-logs/app.log"
                     filePattern="rolling-logs/app-%i.log">
            <Policies>
                <SizeBasedTriggeringPolicy size="10KB"/>
            </Policies>
        </RollingFile>

        <!-- File with JSON layout: structured output for log aggregators -->
        <File name="JsonLogFile" fileName="json-logs/app.json" append="true">
            <JSONTemplateLayout/>
        </File>

    </Appenders>

    <Loggers>

        <!-- Root: catch-all, writes to Console -->
        <Root level="INFO">
            <AppenderRef ref="Console"/>
        </Root>

        <!-- Named loggers: each isolated with additivity=false -->
        <Logger name="LoggerWithFileAppender" level="INFO" additivity="false">
            <AppenderRef ref="LogFile"/>
        </Logger>

        <Logger name="LoggerWithRollingFileAppender" level="INFO" additivity="false">
            <AppenderRef ref="RollingFile"/>
        </Logger>

        <Logger name="LoggerWithJsonFileAppender" level="INFO" additivity="false">
            <AppenderRef ref="JsonLogFile"/>
        </Logger>

        <Logger name="LoggerWithMDC" level="INFO" additivity="false">
            <AppenderRef ref="ConsoleWithMDC"/>
        </Logger>

    </Loggers>

</Configuration>
```

## 7. Summary

| Concept | Detail |
|---|---|
| **Appender** | Sends log events to a destination; a logger can use multiple appenders |
| **Layout** | Formats log events before writing; each appender has one layout |
| **`additivity="false"`** | Stops log events from bubbling up to parent loggers; isolates a logger to its own appenders |
| **Console appender** | Writes to `System.out` or `System.err`; best for development |
| **File appender** | Writes to a single persistent file; `append="true"` accumulates across restarts |
| **RollingFile appender** | Auto-archives files on size or time triggers; `%i` and `%d` in `filePattern` control naming |
| **PatternLayout** | Plain text format via `%` conversion characters; human-readable |
| **JSONTemplateLayout** | Structured JSON output; requires `log4j-layout-template-json` dependency |
| **MDC / ThreadContext** | Thread-local key-value store; values appear in all logs from that thread via `%X{key}` |
| **Lookups** | Reference system properties (`${sys:}`), env vars (`${env:}`), or Log4j internals in config |

**Key takeaways:**
- Appenders and layouts are decoupled — change the format without changing the destination, and vice versa.
- Always set `additivity="false"` on named loggers that have their own appenders to prevent duplicate log output.
- Use `File` for simple persistence; use `RollingFile` in production to prevent unbounded log file growth.
- `PatternLayout` is for human consumption; `JSONTemplateLayout` is for log aggregation tools — the choice depends on who or what is reading the logs.
- MDC is the standard pattern for injecting per-request context (user ID, request ID, session ID) into all log messages on a thread without modifying individual log calls.
