# Introduction to Log4j 2

## 1. Overview

**Apache [Log4j 2](https://logging.apache.org/log4j/2.x/index.html)** is currently one of the most widely adopted logging frameworks in the Java ecosystem. It is a complete rewrite of the original Log4j 1, designed to fix its predecessor's limitations while delivering modern features and significantly better performance.

This lesson covers:
- What Log4j 2 offers over JUL
- Log4j 2's six log levels and their purpose
- Adding Log4j 2 to a Maven project
- Writing first log messages and understanding default behavior

> Log4j 2 may be referred to as either "Log4j 2" or simply "Log4j" in practice. They refer to the same modern framework.

---

## 2. Log4j 2 vs. JUL

Java Util Logging (JUL) is adequate for simple applications, but it has well-known limitations in flexibility, performance, and ecosystem support. Log4j 2 addresses all of them:

| Feature | JUL | Log4j 2 |
|---|---|---|
| **Configuration formats** | `.properties` only | XML, JSON, YAML, `.properties` |
| **Appenders (destinations)** | Console, File only | Console, File, Database, Remote, Message Queues, and more |
| **Filtering** | Level-based only | Level, regex, time range, custom criteria |
| **Asynchronous logging** | No | Yes — writes in a separate thread, application doesn't wait for I/O |
| **Extensibility** | Manual effort | Plugin system for custom appenders, formatters, and filters |
| **Auto-config file discovery** | No — must set system property | Yes — scans classpath automatically |

These capabilities make Log4j 2 suitable for complex enterprise applications where JUL falls short.

## 3. Log Levels

Log4j 2 uses six log levels, listed here from **highest to lowest severity**:

| Level | Method | Purpose |
|---|---|---|
| `FATAL` | `logger.fatal()` | Severe error that may cause the application to terminate entirely |
| `ERROR` | `logger.error()` | Unexpected condition that prevents a specific operation from completing, but does not abort the application |
| `WARN` | `logger.warn()` | Potentially harmful situation the application can recover from, but which may lead to errors if ignored |
| `INFO` | `logger.info()` | Important events during normal operation — milestones, transactions, startup/shutdown |
| `DEBUG` | `logger.debug()` | Detailed internal state information useful for understanding the application's decision-making |
| `TRACE` | `logger.trace()` | Most granular level — tracks execution flow method by method |

**Severity order:**

```
FATAL > ERROR > WARN > INFO > DEBUG > TRACE
```

**Default level:** `ERROR` — only `FATAL` and `ERROR` messages are printed out of the box. All messages below `ERROR` are silently ignored unless the level is reconfigured.

### Log4j 2 vs. JUL Level Mapping

| Log4j 2 | JUL Equivalent |
|---|---|
| FATAL | SEVERE (partial) |
| ERROR | SEVERE |
| WARN | WARNING |
| INFO | INFO |
| DEBUG | FINE |
| TRACE | FINEST |

## 4. Project Setup

### 4.1 Maven Dependency

Log4j 2 is a third-party library and must be added explicitly. Add `log4j-core` to `pom.xml`:

```xml
<dependency>
    <groupId>org.apache.logging.log4j</groupId>
    <artifactId>log4j-core</artifactId>
    <version>2.25.1</version>
</dependency>
```

`log4j-core` provides all the core classes needed to start using the framework — `LogManager`, `Logger`, appenders, and formatters.

> Unlike JUL, which is always available in the JDK, Log4j 2 must be declared as a dependency. The benefit is that it is versioned and can be updated independently of the JDK.

## 5. Writing Log Messages

### 5.1 Obtaining a Logger

```java
package com.baeldung.ljl;

import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;
import org.junit.jupiter.api.Test;

class JavaLoggingUnitTest {

    private static final Logger logger = LogManager.getLogger(JavaLoggingUnitTest.class);

    @Test
    void whenLoggingAtEachLevel_thenDefaultLevelAndAboveArePrinted() {
        logger.fatal("Incompatible database schema version - migration required before startup\n");
        logger.error("Failed to process user payment: invalid card number");
        logger.warn("API response time exceeded 2 seconds threshold");
        logger.info("Monthly newsletter mailed to 100 subscribers");
        logger.debug("User authentication token validated: userId=12345");
        logger.trace("Entering method calculateTotalPrice() with params: items=3, discount=0.15");
    }
}
```

**Key points:**

| Element | Detail |
|---|---|
| `LogManager.getLogger(JavaLoggingUnitTest.class)` | Creates a logger named after the class — standard convention for traceability |
| `private static final Logger` | One logger per class (`static`), immutable reference (`final`), class-private (`private`) |
| Import packages | `org.apache.logging.log4j.LogManager` and `org.apache.logging.log4j.Logger` — both from Log4j 2, not JUL |

> Be careful with imports — `java.util.logging.Logger` (JUL) and `org.apache.logging.log4j.Logger` (Log4j 2) have the same simple name. Always verify the import is from `org.apache.logging.log4j`.

---

### 5.2 Console Output

Running the test with the default configuration produces:

```
2025-09-11T14:13:25.067294Z main FATAL Incompatible database schema version - migration required before startup

2025-09-11T14:13:25.074487Z main ERROR Failed to process user payment: invalid card number
```

Only `FATAL` and `ERROR` appear. `WARN`, `INFO`, `DEBUG`, and `TRACE` are all silently filtered out by the default `ERROR` level threshold.

**What the default format includes:**

| Field | Example |
|---|---|
| Timestamp (ISO-8601) | `2025-09-11T14:13:25.067294Z` |
| Thread name | `main` |
| Level | `FATAL` |
| Message | `Incompatible database schema version - migration required before startup` |

The default format is intentionally minimal — no class name, no method name, no logger name. This is quite different from JUL's default two-line format.

> Notice the `\n` appended to the `fatal()` call. Without it, all log entries run together on a single line in the default format. This is a quirk of Log4j 2's minimal default pattern and will be addressed when we configure a proper layout in a later lesson.

## 6. How Log4j 2 Finds Its Configuration

Unlike JUL — which requires you to explicitly point to a config file via a system property — **Log4j 2 automatically scans the classpath** for a configuration file at startup.

It searches in this order:

```
1. log4j2-test.json / log4j2-test.yaml / log4j2-test.properties / log4j2-test.xml
2. log4j2.json / log4j2.yaml / log4j2.properties / log4j2.xml
3. No file found → fall back to default configuration
```

The `-test` variants take priority, making it easy to use a different (more verbose) configuration during testing without affecting production.

**Default configuration (no file found):**

```
Root logger level : ERROR
Appender          : ConsoleAppender writing to System.out
Layout            : Minimal pattern (timestamp, thread, level, message)
```

This is exactly the behavior we observed in our test output above.

---

## 7. Default Behavior Summary

| Behavior | Default Value | Effect |
|---|---|---|
| Log level | `ERROR` | Only FATAL and ERROR messages appear |
| Output destination | Console (`System.out`) | No file output |
| Format | Minimal (timestamp, thread, level, message) | No class name, method, or logger name |
| Config file discovery | Automatic classpath scan | No system property needed |

---

## 8. Summary

| Concept | Detail |
|---|---|
| **Log4j 2** | Complete rewrite of Log4j 1; modern, high-performance, highly configurable |
| **`log4j-core`** | The Maven dependency that provides all core Log4j 2 classes |
| **`LogManager.getLogger(Class)`** | Standard way to obtain a named logger in Log4j 2 |
| **Six log levels** | FATAL, ERROR, WARN, INFO, DEBUG, TRACE (highest to lowest) |
| **Default level** | ERROR — only FATAL and ERROR are printed without configuration |
| **Default appender** | ConsoleAppender writing to `System.out` |
| **Config discovery** | Automatic classpath scan — no system property required |

**Key takeaways:**
- Log4j 2 is a third-party dependency — add `log4j-core` to `pom.xml` to use it.
- Always import from `org.apache.logging.log4j` — not from `java.util.logging`.
- By default, only `FATAL` and `ERROR` messages appear; all others are filtered out.
- Log4j 2 discovers its configuration file automatically from the classpath — no system property required, unlike JUL.
- The default output format is minimal; layout customization will be covered in upcoming lessons.
