# Introduction to Java Util Logging (JUL)

## 1. Overview

**Java Util Logging (JUL)** is a logging framework built directly into the JDK — no external dependencies required. It lives in the `java.util.logging` package and has been available since JDK 1.4.

This lesson covers:
- Obtaining a logger
- Logging at different levels using both dedicated methods and the generic `log()` method
- Understanding JUL's default behavior and the dual-gate mechanism that controls which messages appear

---

## 2. Project Setup

Since JUL is part of the Java standard library, the `pom.xml` for this module requires **no logging dependencies** — only JUnit for testing:

```xml
<dependencies>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>${junit-jupiter.version}</version>
        <scope>test</scope>
    </dependency>
</dependencies>

<properties>
    <java.version>24</java.version>
    <maven.compiler.release>${java.version}</maven.compiler.release>
    <junit-jupiter.version>5.12.2</junit-jupiter.version>
    <maven-surefire.version>3.5.3</maven-surefire.version>
</properties>
```

This is one of JUL's main advantages — you can start logging immediately in any Java project without adding a single external library.

## 3. Obtaining a Logger

The first step is to get a `Logger` instance using the `Logger.getLogger()` static factory method. Loggers in JUL are identified by a `String` name. By convention, we use the fully-qualified class name so that every log entry clearly indicates where it was generated:

```java
private static final Logger logger = Logger.getLogger(JavaLoggingUnitTest.class.getName());
```

### Why `private static final`?

| Modifier | Reason |
|---|---|
| `private` | The logger belongs to this class only |
| `static` | One logger instance per class — avoids creating a new logger on every object instantiation |
| `final` | Prevents the logger reference from being accidentally reassigned |

> You could technically pass any `String` to `getLogger()`, but using the class name is the standard practice because it makes log entries easy to trace and makes logger configuration straightforward later.

---

## 4. Log Levels

JUL defines its own set of log levels, which map roughly to the levels used in other frameworks:

| JUL Level | Method | Equivalent in Other Frameworks | Purpose |
|---|---|---|---|
| `SEVERE` | `logger.severe()` | ERROR | Serious failures that require immediate attention |
| `WARNING` | `logger.warning()` | WARN | Potential problems that aren't yet critical |
| `INFO` | `logger.info()` | INFO | Normal application events worth recording |
| `CONFIG` | `logger.config()` | — | Configuration-related messages |
| `FINE` | `logger.fine()` | DEBUG | Detailed debugging information |
| `FINER` | `logger.finer()` | DEBUG | More detailed than FINE |
| `FINEST` | `logger.finest()` | TRACE | Most granular debugging information |

**Severity order** (highest to lowest):

```
SEVERE > WARNING > INFO > CONFIG > FINE > FINER > FINEST
```

---

## 5. Logging with Dedicated Methods

The simplest way to log is using the level-specific methods directly:

```java
@Test
void givenLogger_whenLoggingAtDifferentLevels_thenLogsCorrectly() {
    logger.severe("Critical issue occurred while processing task.");
    logger.warning("Potential issue detected in task setup.");
    logger.info("Task processed successfully.");
    logger.config("Configuration loaded correctly.");
    logger.fine("Debug details: task initialization step.");
}
```

**Console output:**

```
Aug 31, 2025 4:29:18 PM com.baeldung.ljl.JavaLoggingUnitTest givenLogger_whenLoggingAtDifferentLevels_thenLogsCorrectly
SEVERE: Critical issue occurred while processing task.

Aug 31, 2025 4:29:18 PM com.baeldung.ljl.JavaLoggingUnitTest givenLogger_whenLoggingAtDifferentLevels_thenLogsCorrectly
WARNING: Potential issue detected in task setup.

Aug 31, 2025 4:29:18 PM com.baeldung.ljl.JavaLoggingUnitTest givenLogger_whenLoggingAtDifferentLevels_thenLogsCorrectly
INFO: Task processed successfully.
```

Notice that `config()` and `fine()` produce **no output**. This is JUL's default behavior — only `INFO` and above are shown. The `CONFIG` and `FINE` messages are filtered out silently.

---

## 6. Logging with the Generic `log()` Method

Instead of the dedicated methods, you can use the generic `log()` method and pass the level explicitly as a `Level` enum value. Both approaches produce identical output:

```java
@Test
void givenLogger_whenLoggingAtDifferentLevelsUsingLog_thenLogsCorrectly() {
    logger.log(Level.SEVERE,  "Critical issue occurred while processing task.");
    logger.log(Level.WARNING, "Potential issue detected in task setup.");
    logger.log(Level.INFO,    "Task processed successfully.");
    logger.log(Level.CONFIG,  "Configuration loaded correctly.");
    logger.log(Level.FINE,    "Debug details: task initialization step.");
}
```

The `log()` method is useful when the log level needs to be determined dynamically at runtime, rather than being hardcoded to a specific method name.

## 7. Default Behavior

By default, JUL applies two layers of filtering before a message reaches the console:

```
Logger (gate 1)
     ↓
  Handler (gate 2)
     ↓
  Console output
```

Both gates must allow a message through for it to appear.

**JUL defaults:**

| Component | Default Level | Effect |
|---|---|---|
| Logger | `INFO` | Messages below INFO are blocked at the logger before reaching any handler |
| ConsoleHandler | `INFO` | Messages below INFO are blocked at the handler even if the logger passed them |

This is why `CONFIG`, `FINE`, `FINER`, and `FINEST` messages never appear without explicit configuration — they are blocked by **both** gates.

---

## 8. Enabling Lower-Level Messages: The Dual-Gate Mechanism

To see messages below `INFO`, **both** the logger level and the handler level must be explicitly lowered. Changing only one is not enough.

```java
@Test
void givenLoggerOnFineLevel_whenLoggingAtDifferentLevels_thenLogsCorrectly() {
    logger.setLevel(Level.FINEST);                           // gate 1: open the logger
    stream(Logger.getLogger("").getHandlers())
        .forEach(handler -> handler.setLevel(Level.FINEST)); // gate 2: open the handler

    logger.severe("Critical issue occurred while processing task.");
    logger.warning("Potential issue detected in task setup.");
    logger.info("Task processed successfully.");
    logger.config("Configuration loaded correctly.");
    logger.fine("Debug details: task initialization step.");
}
```

**Code breakdown:**

| Line | What it does |
|---|---|
| `logger.setLevel(Level.FINEST)` | Opens gate 1 — the logger will now pass all levels to the handler |
| `Logger.getLogger("")` | Retrieves the root logger, which owns the default `ConsoleHandler` |
| `.getHandlers()` | Gets all handlers attached to the root logger |
| `handler.setLevel(Level.FINEST)` | Opens gate 2 — the handler will now write all levels to the console |

**Console output with both gates open:**

```
Aug 31, 2025 4:38:59 PM com.baeldung.ljl.JavaLoggingUnitTest givenLoggerOnFineLevel_whenLoggingAtDifferentLevels_thenLogsCorrectly
SEVERE: Critical issue occurred while processing task.

Aug 31, 2025 4:38:59 PM com.baeldung.ljl.JavaLoggingUnitTest givenLoggerOnFineLevel_whenLoggingAtDifferentLevels_thenLogsCorrectly
WARNING: Potential issue detected in task setup.

Aug 31, 2025 4:39:00 PM com.baeldung.ljl.JavaLoggingUnitTest givenLoggerOnFineLevel_whenLoggingAtDifferentLevels_thenLogsCorrectly
INFO: Task processed successfully.

Aug 31, 2025 4:39:00 PM com.baeldung.ljl.JavaLoggingUnitTest givenLoggerOnFineLevel_whenLoggingAtDifferentLevels_thenLogsCorrectly
CONFIG: Configuration loaded correctly.

Aug 31, 2025 4:39:00 PM com.baeldung.ljl.JavaLoggingUnitTest givenLoggerOnFineLevel_whenLoggingAtDifferentLevels_thenLogsCorrectly
FINE: Debug details: task initialization step.
```

All five messages now appear.

## 9. What the Default Log Format Contains

Each JUL log entry in the default format includes:

```
Aug 31, 2025 4:29:18 PM com.baeldung.ljl.JavaLoggingUnitTest givenLogger_whenLoggingAtDifferentLevels_thenLogsCorrectly
SEVERE: Critical issue occurred while processing task.
```

| Field | Value in example |
|---|---|
| Date and time | `Aug 31, 2025 4:29:18 PM` |
| Logger name (class) | `com.baeldung.ljl.JavaLoggingUnitTest` |
| Method name | `givenLogger_whenLoggingAtDifferentLevels_thenLogsCorrectly` |
| Level | `SEVERE` |
| Message | `Critical issue occurred while processing task.` |

This two-line format is JUL's default. It can be customized by configuring a `Formatter`, which will be covered in a later lesson.

import java.util.logging.Level;

class JavaLoggingUnitTest {

        logger.fine("Debug details: task initialization step.");
    }
}

## 11. Summary

| Concept | Detail |
|---|---|
| **JUL** | Built-in Java logging framework; no external dependencies needed |
| **`Logger.getLogger(className)`** | Standard way to obtain a named logger |
| **`private static final`** | One logger per class, immutable reference, class-private |
| **Log levels** | SEVERE, WARNING, INFO, CONFIG, FINE, FINER, FINEST (high to low) |
| **Dedicated methods** | `severe()`, `warning()`, `info()`, `config()`, `fine()`, `finer()`, `finest()` |
| **Generic method** | `log(Level, message)` — useful when level is determined at runtime |
| **Default behavior** | Only INFO and above are logged; output goes to ConsoleHandler |
| **Dual-gate mechanism** | Both logger level AND handler level must be set to allow lower-level messages through |

**Key takeaways:**
- JUL requires no dependencies — it is always available in any Java project.
- Always obtain loggers using the class name convention for traceability.
- By default, only `SEVERE`, `WARNING`, and `INFO` messages appear.
- To enable `CONFIG`, `FINE`, `FINER`, or `FINEST`, you must lower the level on **both** the logger and its handler — changing only one is not enough.
