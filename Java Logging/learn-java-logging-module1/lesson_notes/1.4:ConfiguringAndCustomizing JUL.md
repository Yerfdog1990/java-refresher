# Configuring and Customizing JUL

## 1. Overview

In the previous lesson we used JUL with its default configuration — INFO and above, output to the console. While that works for simple cases, real projects need more control: different log levels per package, persistent file output, and custom message formats.

This lesson covers how to configure JUL entirely through a `logging.properties` file, without touching application code. Specifically:

- Pointing JUL to a custom `logging.properties` file
- Setting log levels globally and per package
- Configuring `ConsoleHandler` and `FileHandler`
- Customizing output format with `SimpleFormatter` and `XMLFormatter`
- Understanding the limitations of JUL

---

## 2. Project Setup

Like the previous lesson, this module has no external logging dependencies — only JUnit for testing. The only addition is Surefire plugin configuration to pass a system property:

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <version>${maven-surefire.version}</version>
    <configuration>
        <systemPropertyVariables>
            <java.util.logging.config.file>
                ${project.build.testOutputDirectory}/logging.properties
            </java.util.logging.config.file>
        </systemPropertyVariables>
    </configuration>
</plugin>
```

This tells Maven to set the `java.util.logging.config.file` system property when running tests, pointing JUL to our custom `logging.properties` file in `src/test/resources`.

---

## 3. The `logging.properties` File

JUL reads its configuration from a file named `logging.properties` using a simple key-value format. Settings include log levels, handlers, and formatters — all without modifying application code.

### Where JUL looks by default

JDK ships with a global `logging.properties` at `$JAVA_HOME/conf/logging.properties`. Modifying it is impractical because it affects **every Java program on the system**.

### How to use a custom file

Tell JUL where your file is via the system property `java.util.logging.config.file`. This can be set in three ways:

| Method | How |
|---|---|
| Maven/Gradle build file | Via Surefire `systemPropertyVariables` (shown above) |
| Command line | `-Djava.util.logging.config.file=/path/to/logging.properties` |
| Containerized environment | As an environment variable |

Place the file at `src/test/resources/logging.properties` — Maven's Surefire plugin will copy it to the test output directory (`target/test-classes`), which is where `${project.build.testOutputDirectory}` resolves.

> **Important:** When you provide a custom `logging.properties`, the JDK's default global configuration is **completely replaced** — including the default `ConsoleHandler`. You must explicitly re-declare any handlers you need.

## 4. Configuring Log Levels

### 4.1 Global and Package-Level Levels

JUL supports setting log levels at two scopes in `logging.properties`:

```properties
.level = FINEST
com.baeldung.ljl.level = WARNING
```

| Property | Effect |
|---|---|
| `.level = FINEST` | Sets the global (root) level — all loggers inherit this unless overridden |
| `com.baeldung.ljl.level = WARNING` | Overrides the global level for any logger in or under this package |

### 4.2 Logger Hierarchy and Name Matching

JUL resolves which level applies to a logger by matching its name against the **longest prefix** defined in the configuration file. Since logger names follow package/class naming, this forms a natural hierarchy.

**Example:** For a logger named `com.baeldung.ljl.JavaLoggingUnitTest`, JUL checks in this order:

```
com.baeldung.ljl.JavaLoggingUnitTest   ← exact match (not configured)
com.baeldung.ljl                        ← matched → WARNING applies
com.baeldung                            ← not checked (already matched above)
. (root)                                ← fallback if nothing else matched
```

This lets you suppress verbose output from noisy packages while keeping detailed logging elsewhere, all from a single config file.

## 5. Test Class

```java
package com.baeldung.ljl;

import org.junit.jupiter.api.Test;
import java.util.logging.Logger;

class JavaLoggingUnitTest {

    @Test
    void whenMessagesLoggedUsingGlobalAndPackageLogger_thenMessagesLoggedAccordingToConfiguration() {
        Logger globalLogger  = Logger.getGlobal();
        Logger packageLogger = Logger.getLogger(JavaLoggingUnitTest.class.getName());

        logAtAllLevels(globalLogger);
        logAtAllLevels(packageLogger);
    }

    private void logAtAllLevels(Logger logger) {
        String loggerName = logger.getName();
        logger.finest("Sample FINEST log from logger: "  + loggerName);
        logger.finer ("Sample FINER log from logger: "   + loggerName);
        logger.fine  ("Sample FINE log from logger: "    + loggerName);
        logger.config ("Sample CONFIG log from logger: " + loggerName);
        logger.info   ("Sample INFO log from logger: "   + loggerName);
        logger.warning("Sample WARNING log from logger: "+ loggerName);
        logger.severe ("Sample SEVERE log from logger: " + loggerName);
    }
}
```

The test creates two loggers — one global, one for the test class — and calls `logAtAllLevels()` on each, which logs one message at every available JUL level.

## 6. Configuring Handlers

Running the test above with only the level settings in `logging.properties` produces **no output at all** — because the custom file replaced the default global configuration, including the default `ConsoleHandler`. We must declare handlers explicitly.

### 6.1 ConsoleHandler

The `ConsoleHandler` writes log messages to the console. Add it with:

```properties
handlers = java.util.logging.ConsoleHandler
```

**Output after adding ConsoleHandler (with default handler level of INFO):**

```
Oct 20, 2025 7:12:40 PM com.baeldung.ljl.JavaLoggingUnitTest logAtAllLevels
INFO: Sample INFO log from logger: global

Oct 20, 2025 7:12:40 PM com.baeldung.ljl.JavaLoggingUnitTest logAtAllLevels
WARNING: Sample WARNING log from logger: global

Oct 20, 2025 7:12:40 PM com.baeldung.ljl.JavaLoggingUnitTest logAtAllLevels
SEVERE: Sample SEVERE log from logger: global

Oct 20, 2025 7:12:40 PM com.baeldung.ljl.JavaLoggingUnitTest logAtAllLevels
WARNING: Sample WARNING log from logger: com.baeldung.ljl.JavaLoggingUnitTest

Oct 20, 2025 7:12:40 PM com.baeldung.ljl.JavaLoggingUnitTest logAtAllLevels
SEVERE: Sample SEVERE log from logger: com.baeldung.ljl.JavaLoggingUnitTest
```

The global logger shows INFO, WARNING, and SEVERE — but **not** FINEST, FINER, FINE, or CONFIG, even though `.level = FINEST`. This is because the `ConsoleHandler` has its own default level of `INFO`, and it blocks anything below it.

#### The Dual-Gate Mechanism

A message must pass **two independent level checks** before it appears anywhere:

```
Log call
   ↓
Logger level check (gate 1)  — blocks if message < logger's level
   ↓
Handler level check (gate 2) — blocks if message < handler's level
   ↓
Output destination
```

**Both gates must allow the message through.** Configuring only one has no effect on the other.

#### Demonstrating the handler gate

Change the handler level to `SEVERE`:

```properties
java.util.logging.ConsoleHandler.level = SEVERE
```

**Output:**

```
Oct 20, 2025 7:17:33 PM com.baeldung.ljl.JavaLoggingUnitTest logAtAllLevels
SEVERE: Sample SEVERE log from logger: global

Oct 20, 2025 7:17:33 PM com.baeldung.ljl.JavaLoggingUnitTest logAtAllLevels
SEVERE: Sample SEVERE log from logger: com.baeldung.ljl.JavaLoggingUnitTest
```

Even though the global logger allows `FINEST` and the package logger allows `WARNING`, the handler's `SEVERE` gate blocks everything else. Only `SEVERE` gets through.

---

### 6.2 FileHandler

The `FileHandler` writes log messages to a file, providing a persistent record suitable for later analysis and auditing.

Add it alongside `ConsoleHandler`:

```properties
handlers = java.util.logging.ConsoleHandler, java.util.logging.FileHandler

java.util.logging.FileHandler.level   = FINEST
java.util.logging.FileHandler.pattern = application.log
```

| Property | Value | Effect |
|---|---|---|
| `handlers` | Both handlers listed | JUL sends every log event to all listed handlers |
| `FileHandler.level` | `FINEST` | The file handler accepts all messages that pass the logger gate |
| `FileHandler.pattern` | `application.log` | Name of the output file, created in the project root |

After running the test, `application.log` appears in the project root and contains **more messages** than the console — because the `FileHandler` is set to `FINEST` while the `ConsoleHandler` is still at `SEVERE`.

The file output is in **XML format** by default. This is explained in the next section.

## 7. Configuring Formatters

Formatters control how log messages are structured and displayed. Each handler comes with a default formatter already attached — which is why console and file output look different even before we configure formatters explicitly.

### 7.1 SimpleFormatter (Console)

The `ConsoleHandler` uses `SimpleFormatter` by default, producing the human-readable two-line format seen in previous output. We can explicitly declare and customize it:

```properties
java.util.logging.ConsoleHandler.formatter   = java.util.logging.SimpleFormatter
java.util.logging.SimpleFormatter.format     = [%1$tT] %4$s: %5$s%n
```

#### Format placeholder reference

| Placeholder | Meaning | Example output |
|---|---|---|
| `%1$tT` | Timestamp formatted as `HH:mm:ss` | `19:57:20` |
| `%4$s` | Log level | `SEVERE` |
| `%5$s` | Log message | `Sample SEVERE log from logger: global` |
| `%n` | Line separator | (newline) |

**Console output with custom format:**

```
[19:57:20] SEVERE: Sample SEVERE log from logger: global
[19:57:20] SEVERE: Sample SEVERE log from logger: com.baeldung.ljl.JavaLoggingUnitTest
```

The date, class name, and method are removed. The result is much more concise than the default format.

> The explicit `formatter` declaration is not strictly required since `SimpleFormatter` is already the default for `ConsoleHandler` — but it is good practice to be explicit about your configuration.

---

### 7.2 XMLFormatter (File)

The `FileHandler` uses `XMLFormatter` by default. This can be declared explicitly:

```properties
java.util.logging.FileHandler.formatter = java.util.logging.XMLFormatter
```

**Sample `application.log` content:**

```xml
<log>
  <record>
    <date>2025-10-20T14:20:50.073379Z</date>
    <millis>1760970050073</millis>
    <nanos>379000</nanos>
    <sequence>24</sequence>
    <logger>com.baeldung.ljl.JavaLoggingUnitTest</logger>
    <level>SEVERE</level>
    <class>com.baeldung.ljl.JavaLoggingUnitTest</class>
    <method>logAtAllLevels</method>
    <thread>3</thread>
    <message>Sample SEVERE log from logger: com.baeldung.ljl.JavaLoggingUnitTest</message>
  </record>
</log>
```

Every field from the log record is preserved — timestamp to nanosecond precision, source class and method, thread ID, and the message.

**Key difference from `SimpleFormatter`:**

| | SimpleFormatter | XMLFormatter |
|---|---|---|
| **Format** | Plain text | XML |
| **Customizable** | Yes — via `format` property | No — always outputs all fields in fixed schema |
| **Best for** | Human reading / console | Automated log analysis / aggregation |

## 8. Complete `logging.properties`

```properties
# Global log level — all loggers inherit this
.level = FINEST

# Package-level override — restricts com.baeldung.ljl loggers to WARNING and above
com.baeldung.ljl.level = WARNING

# Handlers — both console and file
handlers = java.util.logging.ConsoleHandler, java.util.logging.FileHandler

# ConsoleHandler — only show SEVERE to keep console clean
java.util.logging.ConsoleHandler.level     = SEVERE
java.util.logging.ConsoleHandler.formatter = java.util.logging.SimpleFormatter
java.util.logging.SimpleFormatter.format   = [%1$tT] %4$s: %5$s%n

# FileHandler — capture everything the logger allows
java.util.logging.FileHandler.level     = FINEST
java.util.logging.FileHandler.pattern   = application.log
java.util.logging.FileHandler.formatter = java.util.logging.XMLFormatter
```

## 9. Message Flow Summary

With the complete configuration above, here is exactly what happens when `logAtAllLevels()` is called on the **global logger** (level = `FINEST`):

| Level called | Logger gate (FINEST) | ConsoleHandler gate (SEVERE) | FileHandler gate (FINEST) | Console? | File? |
|---|---|---|---|---|---|
| FINEST | ✓ pass | ✗ block | ✓ pass | No | Yes |
| FINER | ✓ pass | ✗ block | ✓ pass | No | Yes |
| FINE | ✓ pass | ✗ block | ✓ pass | No | Yes |
| CONFIG | ✓ pass | ✗ block | ✓ pass | No | Yes |
| INFO | ✓ pass | ✗ block | ✓ pass | No | Yes |
| WARNING | ✓ pass | ✗ block | ✓ pass | No | Yes |
| SEVERE | ✓ pass | ✓ pass | ✓ pass | **Yes** | **Yes** |

And for the **package logger** (`com.baeldung.ljl`, level = `WARNING`):

| Level called | Logger gate (WARNING) | ConsoleHandler gate (SEVERE) | FileHandler gate (FINEST) | Console? | File? |
|---|---|---|---|---|---|
| FINEST–INFO | ✗ block | — | — | No | No |
| WARNING | ✓ pass | ✗ block | ✓ pass | No | Yes |
| SEVERE | ✓ pass | ✓ pass | ✓ pass | **Yes** | **Yes** |

## 10. Limitations of JUL

Despite being built into the JDK, JUL has several limitations that make it unsuitable for many production applications:

| Limitation | Detail |
|---|---|
| **Inflexible configuration** | Only supports `.properties` format — no YAML or JSON. Requires an explicit system property to locate the file. Configuration changes require an application restart. |
| **Limited handlers** | No built-in support for remote servers, databases, or message queues. Custom handlers require significant implementation effort. |
| **Limited formatters** | Only `SimpleFormatter` (plain text) and `XMLFormatter`. No built-in JSON output or other modern formats. |
| **Synchronous by default** | All logging is synchronous, which can slow the application under high log volume. |
| **Verbose lazy evaluation** | Avoiding unnecessary string building requires more boilerplate compared to modern frameworks. |

These limitations are the main reason why JUL, despite being always available, is not the first choice for large-scale or enterprise applications. Modern alternatives like Logback and Log4j2 address all of these points — which will be covered in upcoming lessons.

## 11. Summary

| Concept | Detail |
|---|---|
| `logging.properties` | Key-value config file that controls JUL behavior without code changes |
| `java.util.logging.config.file` | System property that tells JUL where to find the custom config file |
| `.level` | Sets the global (root) log level |
| `<package>.level` | Overrides the global level for a specific package or class |
| Logger hierarchy | JUL matches the longest prefix of the logger name to find the applicable level |
| `handlers` | Declares which handlers are active; custom file replaces defaults — declare all needed handlers |
| `ConsoleHandler` | Writes to console; default level INFO; default formatter SimpleFormatter |
| `FileHandler` | Writes to file; default formatter XMLFormatter; level and filename configurable |
| Dual-gate mechanism | Both logger level and handler level must allow a message through for it to appear |
| `SimpleFormatter` | Plain text; customizable via `format` property with `%` placeholders |
| `XMLFormatter` | Structured XML; not customizable; captures all fields |

**Key takeaways:**
- Providing a custom `logging.properties` completely replaces the JDK defaults — always declare your handlers explicitly.
- Both the logger level and handler level must pass a message — changing only one has no effect on the other.
- The `FileHandler` defaults to XML output via `XMLFormatter`; the `ConsoleHandler` defaults to plain text via `SimpleFormatter`.
- `SimpleFormatter` output can be customized with the `format` property; `XMLFormatter` cannot be customized.
- JUL is suitable for small projects with basic needs but lacks the flexibility, performance, and ecosystem of modern logging frameworks.
